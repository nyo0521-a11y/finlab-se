これまでClaude Codeを利用して、Claude Codeに推奨された方法でブログを作成してきました。作り始めて2週間は、試行錯誤しながら夢中で運用してきたのですが、1日1記事ペースでアップするようなルーティンができてきたため、現状のブログ運用の仕組みを整理したくなりました。

Claude Codeにこんなことやりたい、あんなことやりたいと相談して、今の仕組みが出来上がりました。その中で、何回もClaude Codeが思ったように動いてくれないこともあり、一度、「仕組みが悪いんじゃないか。全体の仕組みを見直して。」とお願いしたこともあります。このように、Claude Codeに仕組み構築をお願いしていたこともあり、自身でも全体像がどのようになっているのかをぼんやりとしかわかっておらず、今回それをClaude Codeに整理してもらいました。

整理していく中で、以下のような課題が今の仕組みで対処されていることが改めてわかりました。

- 記事を書いてすぐ公開するのは怖い。確認してからアップしたい
- 外出先からでも記事の内容を確認したい
- 「毎朝8時に投稿」みたいな予約投稿が静的サイトでできるか分からない

<strong>結論から言うと、Draft環境と本番環境を分けた2段階フローを持つことで、これらの課題をすべて解決しています。</strong> ファイルの置き場所だけで公開先が変わるシンプルな仕組みなので、一度作ってしまえば運用はほぼ自動です。

なぜこのフローで十分なのか。それは、GitとCloudflare Pagesの組み合わせが「ファイルを動かす＝公開状態を変える」という単純な構造で動いているからです。本記事では、このブログ「金融エンジニアの資産運用実験室」で実際に運用している仕組みをベースに確認していきます。

この記事を読むと、以下のことがわかります。

- Draft確認環境を持つ理由と、その具体的な仕組み
- Draft記事がGoogle検索に引っかからない理由
- 静的サイトで予約投稿を実現する方法

※本記事は個人の運用体験をもとにした仕組み紹介です。構成の詳細は環境によって異なります。

## 登場人物：ブログ運営に出てくるツール・サービス

この仕組みには、聞き慣れない名前がいくつか出てきます。本題に入る前に、それぞれの役割を簡単に整理しておきます。

### GitHub（ギットハブ）

クラウド上のファイル管理サービスです。Wordでいう「変更履歴の保存」を強化したもの、とイメージしてもらうと近いです。「いつ・誰が・何を変えたか」を記録しながらファイルを管理でき、変更をアップロードする操作を「push（プッシュ）」と呼びます。このブログの記事ファイルはすべてGitHubで管理しています。

### Cloudflare Pages（クラウドフレア ページズ）

GitHubと連携して、サイトを自動でビルド（組み立て）・公開してくれるホスティングサービスです。GitHubにファイルをpushするだけで、数十秒後には新しいサイトが公開されます。このブログはDraft（確認用）と本番の2環境をCloudflare Pages上に持っています。

### Hugo（ヒューゴ）

Markdownという書きやすい形式で書いた記事ファイルを、ブラウザで見られるHTML形式に変換するソフトです。「記事を書く作業」と「Webサイトとして見せる作業」を分けてくれる変換エンジンです。

### GitHub Actions（ギットハブ アクションズ）

GitHub上で動く自動化機能です。「毎時0分になったら○○の処理を実行する」「ファイルが更新されたら○○する」といった条件付き・定時実行の仕組みを設定できます。このブログの予約投稿機能はGitHub Actionsで動いています。

### Claude Code（クロード コード）

AnthropicのAI「Claude」をターミナル（コマンド操作画面）から使えるツールです。記事の執筆・修正はもちろん、GitHub Actionsのワークフロー設定ファイルのような技術的な実装もClaude Codeに依頼しながら作ってきました。このブログ運営の実質的な共同作業者です。

このブログの記事は、Claude Codeにベースとなる下書きを作成してもらい、そこに私自身の経験や知見をClaude Codeと対話しながら追加・修正してもらう形で作成しています。

## 全体フロー：2つのCloudflare Pages環境

<strong>このブログは「Draft環境」と「本番環境」の2つのCloudflare Pages環境で動いています。</strong>

2つの環境の役割はシンプルです。

| 環境 | URL | 用途 |
|---|---|---|
| Draft環境 | draft.finlab-se.pages.dev | 確認用（外部公開しない） |
| 本番環境 | finlab-se.com | 実際に公開するサイト |

GitHubのmainブランチにpushすると、両方の環境が自動でビルドされます。どちらに記事が出るかは、<strong>ファイルをどのフォルダに置くかで決まります。</strong>

- `content/drafts/` に置いた記事 → Draft環境にだけ表示される
- `content/posts/` に置いた記事 → 本番環境に表示される

本番アップしたいときは `git mv` コマンドでフォルダを移動してpushするだけです。コピーではなく「移動」なので、同期ずれが起きません。

## なぜDraft環境が必要か

### ローカルでの確認には限界がある

Hugoはローカルで `hugo server` コマンドを使えばプレビューできます。ただし、<strong>Cloudflare Pages固有の設定はローカルでは再現されません。</strong>

たとえばOGP（SNSでシェアされたときのサムネイル・タイトル表示）はローカルでは確認できません。「ローカルでは正しく見えていたのに、本番アップしたら壊れていた」を防ぐためにも、本番に近い環境での確認は重要です。

### スマホで確認できるのが大きい

もうひとつ大きい理由が、<strong>外出先でもスマホで確認できることです。</strong>

ローカルサーバーは自宅のPCでしか動きません。「電車の中でスマホから見たらどう見える？」を確認するためには、URLがあるWeb環境が必要です。Draft環境はURLが固定されているので、どこからでもスマホブラウザで確認できます。

記事のレイアウト崩れや、スマホで見たときの読みやすさを本番アップ前に確認できるのは、思っていた以上に便利です。

---

## Draft環境はGoogle検索に引っかからない

Draft環境はURLを知っていれば誰でも見られますが、<strong>Google検索には出てきません。</strong>

仕組みはシンプルで、Draft環境の `robots.txt` に以下を設定しています。

```
User-agent: *
Disallow: /
```

これはGoogleなどの検索エンジンのクローラー（自動巡回プログラム）に対して「このサイトを読み込まないでください」と伝える記述です。この指示に従ったクローラーはページをインデックス（検索データベースへ登録）しません。

なぜこれが必要かというと、<strong>「本番より先にDraftの内容がGoogle検索に出てしまう」リスクを防ぐためです。</strong>

同じ内容のページが2つのURLに存在すると、Googleから重複コンテンツとみなされてSEO評価に悪影響が出る場合があります。また、確認中の未完成の状態が外部に漏れることも避けたい。`robots.txt` での制御はそのための最低限の対策です。

## 公開日時の予約投稿はどう実現しているか

静的サイトには、ブログサービス（はてなブログ、noteなど）のような「予約投稿」機能がありません。代わりに、<strong>frontmatterのdateフィールドとGitHub Actionsのcron機能を組み合わせて予約投稿を実現しています。</strong>

### frontmatterのdateで「いつ公開するか」を指定する

ブログ記事のMarkdownファイルの先頭には、frontmatterという設定ブロックがあります。

```yaml
---
title: "記事タイトル"
date: 2026-05-01T08:00:00+09:00
---
```

Hugoはこの `date` フィールドを「記事の公開日時」として扱います。

- `date` が現在時刻より<strong>過去</strong>の場合：通常通り本番アップの対象になる
- `date` が現在時刻より<strong>未来</strong>の場合：Hugoのビルド時に除外され、本番に出ない

つまり `date: 2026-05-01T08:00:00+09:00` と書いておけば、5月1日8時になるまでは公開されません。

### GitHub ActionsのCronが1時間ごとに自動チェック

「date が過去になったら自動で本番アップする」処理を、GitHub Actionsのcron機能で動かしています。

GitHubには「定時に処理を実行する」スケジュール実行機能があり、これをcronと呼びます。設定は以下のような記述で行います（これはYAML形式のワークフローファイルです）。

```yaml
on:
  schedule:
    - cron: '0 * * * *'  # 毎時0分に実行
```

このワークフローが1時間に1回、以下の処理を自動で行います。

1. `content/drafts/` の記事を全件チェック
2. `date` フィールドが現在時刻より過去になった記事を検知
3. 該当記事を `content/posts/` へ移動（git mv）
4. 自動でcommit・pushを実行
5. pushを受けてCloudflare Pagesが本番ビルドを実行
6. 記事が本番に公開される

<strong>「毎朝8時に公開」と設定しておけば、最大で8時59分までには自動で本番に上がります。</strong> 1時間に1回のチェックなので、最大59分のずれがありますが、個人ブログの運用では十分な精度です。

### なぜこの仕組みにしたか

定期的にコンテンツが更新されるブログは、読者が「また来よう」と思いやすくなります。「毎週火曜に更新」のような定期更新をするためには、予約投稿が必要でした。

静的サイトをそのまま使うとこの仕組みがないため、GitHub Actionsで自作することにしました。

## Claude Codeとの組み合わせ

記事執筆・修正はClaude Codeで行うことが多く、このDraft→本番フローとの相性がよいと感じています。

実際のサイクルは以下のような流れです。

1. Claude Codeに記事の下書きを作成・修正してもらう
2. mainブランチにpushしてDraft環境に反映させる
3. スマホで `draft.finlab-se.pages.dev` にアクセスして確認する
4. 気になる点をClaude Codeに伝えて修正してもらう
5. 再pushで確認。問題なければ `git mv` で本番フォルダへ移動して公開

<strong>GitHub Actionsのワークフローファイル自体もClaude Codeで作成しました。</strong> YAMLの文法や条件分岐の書き方を自分で調べなくても、「こういう処理を1時間ごとに動かしたい」と伝えると実装してくれます。

静的サイトを選ぶと「自由に設定できる反面、仕組みを自分で作る必要がある」という面があります。その部分をClaude Codeが補ってくれることで、エンジニアでなくても自分専用の運用フローを作れるようになっています。

## まとめ

このブログの公開フローを整理すると、以下のようになります。

- GitHubのmainブランチへのpushで、Draftと本番の両方が自動ビルドされる
- `content/drafts/` に置いた記事はDraft環境に表示され、`content/posts/` に置いた記事が本番に表示される
- Draft環境は `robots.txt` でクローラーをブロックし、検索エンジンには出ない
- frontmatterの `date` フィールドを未来日時にすることで予約投稿を実現できる
- GitHub ActionsのCronが1時間ごとにチェックし、dateが過去になった記事を自動で本番へ移動・公開する

「書いたらすぐ公開」ではなく「確認してから公開」できる安心感は、継続的なブログ運営にとって思っていた以上に重要でした。

まずはDraft環境を作るところから試してみると、自分のブログ運営に合った公開フローが見えてくると思います。何かの参考になれば幸いです。
